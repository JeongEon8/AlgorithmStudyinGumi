# [PGS - Lv2] 01_2개 이하로 다른 비트

## ⏰**time**

34분

## :pushpin: **Algorithm**

* 비트마스킹
* 비트 연산
* 수학

## ⏲️**Time Complexity**

$O(N)$
`N`은 `numbers` 배열의 길이이다.
각 숫자마다 짝수 여부를 확인하고, 몇 번의 비트 연산만 수행하므로 숫자 하나당 $O(1)$이 걸린다.

따라서 전체 시간 복잡도는 $O(N)$이다.

## :round_pushpin: **Logic**

1. 각 숫자 `x`보다 크면서, 이진수 표현에서 다른 비트가 2개 이하인 가장 작은 수를 찾아야 한다.

2. 숫자가 짝수라면 가장 오른쪽 비트가 `0`이다.

   따라서 `1`을 더하면 마지막 비트만 `0 → 1`로 바뀌므로, 다른 비트가 1개뿐인 가장 작은 수가 된다.

   ```java
   if ((num & 1) == 0) {
       answer[i] = num + 1;
   }
   ```

3. 숫자가 홀수라면 가장 오른쪽 비트가 `1`이다.

   홀수에서 조건을 만족하는 가장 작은 수를 만들려면, 오른쪽에서 가장 가까운 `0`을 `1`로 바꾸고 그 바로 오른쪽의 `1`을 `0`으로 바꿔야 한다.

   예를 들어 `11`은 다음과 같다.

   ```text
   11 = 1011
   ```

   오른쪽에서 가장 가까운 `0`을 `1`로 바꾸고, 바로 오른쪽의 `1`을 `0`으로 바꾸면 다음 수가 된다.

   ```text
   1011
   1101
   ```

   결과는 `13`이다.

4. `~num`은 `num`의 비트를 반전한다.

   따라서 원래 숫자에서 `0`인 비트는 `~num`에서 `1`이 된다.

5. `num + 1`은 홀수의 오른쪽 연속된 `1`들을 `0`으로 바꾸고, 그 왼쪽의 첫 번째 `0`을 `1`로 바꾼다.

6. 다음 비트 연산을 사용하면 오른쪽에서 가장 가까운 `0`의 자릿값만 구할 수 있다.

   ```java
   long lastZero = (~num) & (num + 1);
   ```

   예를 들어 `num = 11`일 때:

   ```text
   num        = 1011
   num + 1    = 1100
   ~num       = ...0100
   lastZero   = 0100
   ```

   따라서 `lastZero`는 `4`가 된다.

7. `lastZero` 위치의 `0`을 `1`로 바꾸기 위해 `lastZero`를 더한다.

   그리고 바로 오른쪽 비트의 `1`을 `0`으로 바꾸기 위해 `lastZero >> 1`을 뺀다.

   ```java
   answer[i] = num + lastZero - (lastZero >> 1);
   ```

8. 모든 숫자에 대해 계산한 결과 배열을 반환한다.

   ```java
   return answer;
   ```

## :black_nib: **Review**

* 처음에는 숫자를 이진 문자열로 바꾼 뒤 오른쪽에서 가장 가까운 `0`을 찾아 수정하는 방식으로 접근했다.
* 짝수는 마지막 비트가 `0`이므로 단순히 `1`을 더하면 정답이 된다는 점을 이용할 수 있었다.
* 홀수는 오른쪽에서 가장 가까운 `0`과 그 오른쪽의 `1`을 서로 바꾸는 형태라는 규칙이 핵심이었다.
* `(~num) & (num + 1)`을 사용하면 오른쪽에서 가장 가까운 `0`의 자릿값을 한 번에 구할 수 있었다.
* 문자열 변환과 파싱 없이 비트 연산만 사용하므로 각 숫자를 $O(1)$에 처리할 수 있었다.
